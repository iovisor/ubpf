## 1. Overview

This document specifies the complete mapping from BPF ISA instructions to x86-64 machine code
as implemented by the uBPF JIT compiler backend. It covers register allocation, instruction
translation, calling conventions, security hardening, and platform differences.

The JIT compiler translates eBPF bytecode in a single pass, emitting x86-64 machine code into
a caller-provided buffer. After code emission, a fixup pass resolves all forward references
(jumps, loads, LEAs, local calls). The compiled code is position-independent within its
contiguous code+data buffer: all internal references use RIP-relative addressing, so the
buffer can be relocated as a unit before execution.

**Entry point:** `ubpf_translate_x86_64()` (`vm/ubpf_jit_x86_64.c:2628`)
**Compilation pipeline:**
1. `initialize_jit_state_result()` — allocate tracking structures (`vm/ubpf_jit_support.c:65`)
2. `translate()` — emit prologue, instruction code, epilogue, retpoline, data tables (`vm/ubpf_jit_x86_64.c:1699`)
3. `resolve_patchable_relatives()` — patch all placeholder offsets (`vm/ubpf_jit_x86_64.c:2516`)

**JIT'd code memory layout** (documented at `vm/ubpf_jit_x86_64.c:1603–1629`):
```
buffer:          Machine code (prologue + instructions + epilogue)
                 Retpoline gadget
                 External dispatcher function pointer (8 bytes, may be NULL)
                 Helper function pointer table (MAX_EXT_FUNCS × 8 bytes)
buffer+offset:   End
```

## 2. Register Mapping

The JIT maps 11 BPF registers (R0–R10) to x86-64 general-purpose registers.
The mapping differs between System V and Microsoft x64 ABIs.

> **Source:** `vm/ubpf_jit_x86_64.c:68–131`

### 2.1 System V ABI (Linux/macOS)

| BPF Reg | x86-64 Reg | Role | Volatility (x86-64) |
|---------|-----------|------|---------------------|
| R0 | RAX | Return value | Volatile (caller-saved) |
| R1 | RDI | Arg 1 / context ptr | Volatile |
| R2 | RSI | Arg 2 | Volatile |
| R3 | RDX | Arg 3 | Volatile |
| R4 | R10 | Arg 4 (mapped to volatile) | Volatile |
| R5 | R8 | Arg 5 | Volatile |
| R6 | RBX | Callee-saved 1 | Non-volatile |
| R7 | R12 | Callee-saved 2 | Non-volatile |
| R8 | R13 | Callee-saved 3 | Non-volatile |
| R9 | R14 | Callee-saved 4 | Non-volatile |
| R10 | R15 | Frame pointer (read-only) | Non-volatile |

**Source:** `vm/ubpf_jit_x86_64.c:115–129`

**Platform non-volatile registers saved:** RBP, RBX, R12, R13, R14, R15 (6 registers)
(`vm/ubpf_jit_x86_64.c:111`)

**Platform parameter registers (calling out):** RDI, RSI, RDX, RCX, R8, R9
(`vm/ubpf_jit_x86_64.c:114`)

### 2.2 Microsoft x64 ABI (Windows)

| BPF Reg | x86-64 Reg | Role | Volatility (x86-64) |
|---------|-----------|------|---------------------|
| R0 | RAX | Return value | Volatile |
| R1 | R10 | Arg 1 (mapped to volatile) | Volatile |
| R2 | RDX | Arg 2 | Volatile |
| R3 | R8 | Arg 3 | Volatile |
| R4 | R9 | Arg 4 | Volatile |
| R5 | R12 | Arg 5 | Non-volatile |
| R6 | RBX | Callee-saved 1 | Non-volatile |
| R7 | RDI | Callee-saved 2 | Non-volatile |
| R8 | RSI | Callee-saved 3 | Non-volatile |
| R9 | R14 | Callee-saved 4 | Non-volatile |
| R10 | R15 | Frame pointer (read-only) | Non-volatile |

**Source:** `vm/ubpf_jit_x86_64.c:95–109`

**Platform non-volatile registers saved:** RBP, RBX, RDI, RSI, R12, R13, R14, R15 (8 registers)
(`vm/ubpf_jit_x86_64.c:92`)

**Platform parameter registers (calling out):** RCX, RDX, R8, R9
(`vm/ubpf_jit_x86_64.c:94`)

**Note on R15:** On both platforms, R15 is always mapped to BPF R10 (the frame pointer).
Comment at `vm/ubpf_jit_x86_64.c:108`: *"Until further notice, r15 must be mapped to eBPF register r10"*

**Note on R12:** R12 is callee-saved on both platforms and IS included in `platform_nonvolatile_registers`. However, the source comment at `vm/ubpf_jit_x86_64.c:74–76` states: *"R12 is special and we are \*not\* using it"* — this means R12 is not used as a BPF register mapping on System V (it does not appear in the System V `register_map[]`). On Windows, R12 IS mapped to BPF R5. On both platforms, R12 is saved/restored as part of the non-volatile register set in the prologue/epilogue.

### 2.3 Scratch Registers

| Register | Constant | Usage |
|----------|----------|-------|
| R11 | `VOLATILE_CTXT` (= 11) | Stores context pointer during helper calls. Used as temporary in constant blinding. (`vm/ubpf_jit_x86_64.c:58`) |
| R10 | `RCX_ALT` (= 10) | Alternative for RCX on Windows; used as scratch during external calls. (`vm/ubpf_jit_x86_64.c:89`) |
| RCX | 1 | Used for shift operations (x86 requires shift count in CL). Saved/restored as needed. |
| RAX, RDX | 0, 2 | Used implicitly by MUL/DIV/MOD. Saved/restored around these operations. |

**Zero-extension handling** (`vm/ubpf_jit_x86_64.c:2442–2444`):
After every ALU32 instruction (except byte-swap `0xd0`), the JIT emits:
```asm
and dst_32, 0xFFFFFFFF    ; emit_truncate_u32: opcode 81 /4 id
```
This explicitly zeroes the upper 32 bits of the destination register, conforming to RFC 9669
§4.1 which requires ALU32 results to be zero-extended to 64 bits.

**Source:** `emit_truncate_u32()` at `vm/ubpf_jit_x86_64.c:337–339`

### 3.3 Signed Arithmetic (SDIV, SMOD)

**Cross-ref:** REQ-UBPF-ISA-DIV-003 (Signed Division), REQ-UBPF-ISA-DIV-004 (Signed Modulo)

Signed division and modulo are handled by `emit_muldivmod()` (`vm/ubpf_jit_x86_64.c:1270–1472`).
The `offset` field of the BPF instruction distinguishes signed (`offset == 1`) from unsigned
(`offset == 0`).

**Common sequence for MUL/DIV/MOD:**
```asm
; Save RAX and RDX if they are not the destination
push rax              ; if dst != RAX (line 1293)
push rdx              ; if dst != RDX (line 1297)

; Load divisor/multiplier
mov rcx, src          ; register variant (line 1303)
; — or —
mov rcx, imm          ; immediate variant via EMIT_LOAD_IMM (line 1301)

; Load dividend
mov rax, dst          ; (line 1307)
```

**Division-by-zero handling (div/mod only, lines 1316–1351):**
```asm
test rcx, rcx         ; Check if divisor is zero
push rax              ; Save dividend (mod only, line 1327)
pushfq                ; Save flags (line 1330)
mov rdx, 1            ; Prepare substitute divisor
cmove rcx, rdx        ; Set divisor to 1 if zero (line 1334–1337)
```

**Signed vs. unsigned setup:**
- **Unsigned:** `xor edx, edx` — zero-extend RAX into RDX:RAX (line 1349)
- **Signed 64-bit:** `cqo` (`48 99`) — sign-extend RAX into RDX:RAX (lines 1342–1343)
- **Signed 32-bit:** `cdq` (`99`) — sign-extend EAX into EDX:EAX (line 1345)

**Signed overflow guard (INT_MIN / -1)** (`vm/ubpf_jit_x86_64.c:1353–1412`):
Per RFC 9669, `INT_MIN / -1` must produce `INT_MIN` (wrapping). On x86-64, `IDIV` would
raise #DE. The JIT emits:
```asm
cmp rcx, -1           ; Check if divisor is -1
jne skip
; Check if dividend is INT_MIN (0x8000000000000000 for 64-bit)
mov r11, 0x8000000000000000
cmp rax, r11
jne skip
; For SDIV: result already in RAX (INT_MIN), skip IDIV
; For SMOD: xor edx, edx (result is 0), skip IDIV
jmp after_idiv
skip:
```

**Actual multiply/divide instruction:**
```asm
; REX.W prefix if 64-bit (line 1414)
F7 /4               ; mul rcx   (for MUL)
F7 /6               ; div rcx   (for unsigned DIV/MOD)
F7 /7               ; idiv rcx  (for signed SDIV/SMOD)
```
Source: `vm/ubpf_jit_x86_64.c:1422–1423`

**Result selection:**
- MUL, DIV: result in RAX → `mov dst, rax` (line 1468)
- MOD: result in RDX → `mov dst, rdx` (line 1462)

**Division-by-zero result:**
- DIV: `cmove rax, 0` — result is 0 (lines 1437–1446)
- MOD: `cmove rdx, saved_dividend` — result is dividend (lines 1448–1457)

### 3.5 Byte Swap Operations

**Cross-ref:** REQ-UBPF-ISA-SWAP-001 (Endianness Conversion), REQ-UBPF-ISA-SWAP-002 (Unconditional Byte Swap)
x86-64 is already little-endian; LE is a no-op except for truncation:

| imm | x86-64 Sequence |
|-----|-----------------|
| 16 | `and dst_32, 0xFFFF` |
| 32 | `and dst_32, 0xFFFFFFFF` |
| 64 | (no-op) |

**BE (Big-Endian, `EBPF_OP_BE`, `vm/ubpf_jit_x86_64.c:1954–1967`):**

| imm | x86-64 Sequence |
|-----|-----------------|
| 16 | `66 C1 /0 08` (`rol dst_16, 8`) + `and dst_32, 0xFFFF` |
| 32 | `0F C8+rd` (`bswap dst_32`) |
| 64 | `REX.W 0F C8+rd` (`bswap dst_64`) |

**BSWAP (unconditional byte swap, `EBPF_OP_BSWAP`, `vm/ubpf_jit_x86_64.c:1969–1989`):**

| imm | x86-64 Sequence |
|-----|-----------------|
| 16 | `66 C1 /0 08` (`rol dst_16, 8`) + `REX.W 81 /4 id 0xFFFF` (`and dst_64, 0xFFFF`) |
| 32 | `0F C8+rd` (`bswap dst_32`) + `89 /r` (`mov dst_32, dst_32` — zero-extend) |
| 64 | `REX.W 0F C8+rd` (`bswap dst_64`) |

**Displacement encoding** (`emit_modrm_and_displacement`, `vm/ubpf_jit_x86_64.c:239–270`):
- `offset == 0` and base not RSP/RBP/R12/R13: ModRM with mod=00 (no displacement)

### 3.9 64-bit Immediate (LDDW)

**Cross-ref:** REQ-UBPF-ISA-LDDW-001 (Basic 64-bit Immediate Load)

**Source:** `vm/ubpf_jit_x86_64.c:2342–2347`

The LDDW instruction spans two BPF instruction slots. The JIT combines both immediates:
```c
struct ebpf_inst inst2 = ubpf_fetch_instruction(vm, ++i);
uint64_t imm = (uint32_t)inst.imm | ((uint64_t)inst2.imm << 32);
```

The combined 64-bit immediate is loaded via `emit_load_imm()`:
- If fits in 32-bit sign-extended: `REX.W C7 /0 id` — `mov dst, imm32` (7 bytes)
- Otherwise: `REX.W B8+rd io` — `movabs dst, imm64` (10 bytes)

| `JSET32_REG` | `test dst_32, src_32` | `0F 85` | 2222–2223 |
| `JNE32_IMM` / `JNE32_REG` | `cmp dst_32, {imm\|src_32}` | `0F 85` | 2226–2232 |
| `JSGT32_IMM` / `JSGT32_REG` | `cmp dst_32, {imm\|src_32}` | `0F 8F` | 2234–2240 |
| `JSGE32_IMM` / `JSGE32_REG` | `cmp dst_32, {imm\|src_32}` | `0F 8D` | 2242–2248 |
| `JSLT32_IMM` / `JSLT32_REG` | `cmp dst_32, {imm\|src_32}` | `0F 8C` | 2250–2256 |
| `JSLE32_IMM` / `JSLE32_REG` | `cmp dst_32, {imm\|src_32}` | `0F 8E` | 2258–2263 |

**`emit_cmp`** (`vm/ubpf_jit_x86_64.c:388`): `REX.W 39 /r` — 64-bit `cmp dst, src`
**`emit_cmp_imm32`** (`vm/ubpf_jit_x86_64.c:377`): `REX.W 81 /7 id` — 64-bit `cmp dst, imm32`
**`emit_cmp32`** (`vm/ubpf_jit_x86_64.c:394`): `39 /r` — 32-bit `cmp dst_32, src_32`
**`emit_cmp32_imm32`** (`vm/ubpf_jit_x86_64.c:383`): `81 /7 id` — 32-bit `cmp dst_32, imm32`

### 3.11 Atomic Operations

**Cross-ref:** REQ-UBPF-ISA-ATOM-001 (Simple Atomics), REQ-UBPF-ISA-ATOM-002 (Complex Atomics)

All atomic operations use the `LOCK` prefix (`F0`). Source: `vm/ubpf_jit_x86_64.c:1032–1267`.

**64-bit atomics (`EBPF_OP_ATOMIC_STORE`, lines 2348–2389):**

| BPF imm | Fetch? | x86-64 Sequence | Function | Line |
|---------|--------|-----------------|----------|------|
| ADD | No | `lock add [dst+off], src` | `emit_atomic_add64` | 1150 |
| ADD\|FETCH | Yes | CAS loop (see below) | `emit_atomic_fetch_add64` | 1222 |
| OR | No | `lock or [dst+off], src` | `emit_atomic_or64` | 1174 |
| OR\|FETCH | Yes | CAS loop | `emit_atomic_fetch_or64` | 1246 |
| AND | No | `lock and [dst+off], src` | `emit_atomic_and64` | 1162 |
| AND\|FETCH | Yes | CAS loop | `emit_atomic_fetch_and64` | 1234 |
| XOR | No | `lock xor [dst+off], src` | `emit_atomic_xor64` | 1186 |
| XOR\|FETCH | Yes | CAS loop | `emit_atomic_fetch_xor64` | 1258 |
| XCHG | — | `lock xchg [dst+off], src` | `emit_atomic_exchange64` | 1210 |
| CMPXCHG | — | `lock cmpxchg [dst+off], src` | `emit_atomic_compare_exchange64` | 1198 |

**32-bit atomics (`EBPF_OP_ATOMIC32_STORE`, lines 2391–2434):**
Same structure as 64-bit but without REX.W prefix. Additionally:
- `XCHG32`: followed by `emit_truncate_u32(src)` (line 2424)
- `CMPXCHG32`: followed by `emit_truncate_u32(map_register(0))` to truncate R0/RAX (line 2428)

**Non-fetch atomic ALU encoding** (`emit_atomic_alu`, `vm/ubpf_jit_x86_64.c:1032–1038`):
```asm
F0                    ; LOCK prefix
[REX.W]               ; if 64-bit
{01|09|21|31}         ; ADD/OR/AND/XOR opcode
ModRM + displacement  ; [dst + offset]
```

**Fetch-ALU emulation via CAS loop** (`emit_atomic_fetch_alu`, `vm/ubpf_jit_x86_64.c:1096–1147`):
Although x86-64 has `LOCK XADD` for atomic fetch-add, it lacks atomic fetch-or, fetch-and, and fetch-xor instructions. For uniformity, the JIT emulates ALL fetch-ALU variants (including fetch-add) using a `LOCK CMPXCHG` compare-and-swap loop:
```asm
push rax                    ; Save RAX (if src != RAX)
mov rax, [dst+off]          ; Load original value
loop:
  mov rcx, rax              ; Copy original to RCX
  {add|or|and|xor} rcx, src ; Apply operation
  lock cmpxchg [dst+off], rcx  ; Try to swap: if [dst+off]==RAX, write RCX
  jne loop                  ; Retry if value changed (ZF=0)
mov src, rax                ; Return old value in src register
pop rax                     ; Restore RAX
```

**Atomic exchange** (`emit_atomic_exchange`, `vm/ubpf_jit_x86_64.c:1074–1080`):
```asm
F0 [REX.W] 87 /r     ; lock xchg [dst+off], src
```

**Atomic compare-exchange** (`emit_atomic_cmp_exch_with_rax`, `vm/ubpf_jit_x86_64.c:1054–1061`):
```asm
F0 [REX.W] 0F B1 /r  ; lock cmpxchg [dst+off], src
; Implicitly compares [dst+off] with RAX (BPF R0)
; If equal: [dst+off] ← src; RAX unchanged
; If not equal: RAX ← [dst+off]
```

### 3.12 CALL Instructions

**Source:** `vm/ubpf_jit_x86_64.c:2265–2281`

Two call types are distinguished by the `src` field of the instruction:

| src | Type | Handler |
|-----|------|---------|
| 0 | External helper call | `emit_dispatched_external_helper_call()` |
| 1 | Local function call | `emit_local_call()` |

See §6 (Helper Function Dispatch) and §7 (Local Function Calls) for details.

**Unwind-on-success:** If the called helper's index matches `vm->unwind_stack_extension_index`,
the JIT emits a check after the call (lines 2271–2276):
```asm
cmp R0_mapped, 0      ; Check return value
je  exit_loc           ; If zero, exit the BPF program
```

### 3.13 EXIT Instruction

**Source:** `vm/ubpf_jit_x86_64.c:2282–2291`

The EXIT instruction pops the local function's stack usage from the host stack and returns
to the caller (either the parent BPF function or the JIT prologue's fake `call`):

```asm
add rsp, 8            ; Pop the 8-byte stack-usage value (REX.W 81 /0 08)
ret                   ; Return to caller (C3)
```

The 8-byte `add rsp` maintains the invariant that the top of the host stack always holds
the current function's stack usage (see §4).

## 4. Function Prologue and Epilogue

### 4.1 BasicJitMode

**Source:** `vm/ubpf_jit_x86_64.c:1700–1775` (prologue), `vm/ubpf_jit_x86_64.c:2487–2512` (epilogue)

**Prologue sequence:**
```asm
; 1. Save platform non-volatile registers
push rbp
push rbx
; (System V: push r12, r13, r14, r15 — 6 total)
; (Windows:  push rdi, rsi, r12, r13, r14, r15 — 8 total)

; 2. Move first parameter to BPF R1
mov BPF_R1, param_reg[0]     ; if BPF_R1 != param_reg[0]

; 3. Save context pointer in VOLATILE_CTXT (R11)
mov r11, param_reg[0]

; 4. Align stack to 16 bytes (if even number of pushes)
sub rsp, 8                    ; Only if _countof(nonvolatile_regs) is even

; 5. Save RSP in RBP for later restoration
mov rbp, rsp

; 6. Set BPF R10 (frame pointer) to current RSP
mov r15, rsp

; 7. Allocate eBPF stack space
sub rsp, UBPF_EBPF_STACK_SIZE    ; Default: MAX_CALL_DEPTH * 512

; 8. (Windows only) Allocate home register space
sub rsp, 32                       ; 4 × 8 bytes

; 9. Fake call to establish return point
call +5                           ; Push return address, skip next instruction
jmp exit_loc                      ; Jumped over initially; reached on final EXIT
; ... BPF instruction code begins here ...
```

**Stack alignment note** (line 1719–1731): After the `call` that enters the JIT'd code,
the stack is misaligned by 8 bytes (return address). The non-volatile register pushes
further adjust alignment. If an even number of registers were pushed, an extra 8 bytes
is subtracted to regain 16-byte alignment.

### 4.2 ExtendedJitMode

**Source:** `vm/ubpf_jit_x86_64.c:1747–1751`

In ExtendedJitMode, the caller provides the stack via parameters 3 and 4:
```asm
; Instead of steps 6-7, use caller-provided stack:
mov r15, param_reg[2]             ; stack_ptr (3rd parameter)
add r15, param_reg[3]             ; stack_ptr + stack_len (R10 points to top)
```

Function signature: `uint64_t fn(void* mem, size_t mem_len, uint8_t* stack, size_t stack_len)`

Parameter register mapping:
- System V: `param_reg[2]` = RDX, `param_reg[3]` = RCX
- Windows: `param_reg[2]` = R8, `param_reg[3]` = R9

### 4.3 Epilogue (Both Modes)

**Source:** `vm/ubpf_jit_x86_64.c:2487–2512`

```asm
exit_loc:
; 1. Move BPF R0 to RAX (return value)
mov rax, BPF_R0               ; if BPF_R0 != RAX

; 2. Restore RSP from RBP (deallocates all stack space)
mov rsp, rbp

; 3. Remove alignment padding (if even number of pushes)
add rsp, 8                    ; Only if _countof(nonvolatile_regs) is even

; 4. Restore platform non-volatile registers (reverse order)
pop r15
pop r14
; (System V: pop r13, r12, rbx, rbp — 6 total)
; (Windows:  pop r15, r14, r13, r12, rsi, rdi, rbx, rbp — 8 total)

; 5. Return
ret                            ; C3
```

### 4.4 Per-Function Prolog (Local Functions)

**Source:** `vm/ubpf_jit_x86_64.c:1825–1843`

At the start of every BPF local function (including the entry point at PC 0), the JIT emits
a prolog that pushes the function's stack usage onto the host stack:
```asm
sub rsp, 8                    ; Allocate 8 bytes on host stack
mov qword [rsp], stack_usage  ; Store this function's eBPF stack usage
; Encoding: 48 C7 04 24 <imm32>
```

This maintains the invariant that `[rsp]` always contains the current eBPF function's
stack usage, which is consumed by local call and EXIT sequences.

The prolog size is recorded in `state->bpf_function_prolog_size` for accurate local
call offset calculation (line 1838).

## 5. Security Features

### 5.1 Constant Blinding

**Source:** `vm/ubpf_jit_x86_64.c:462–695`, `vm/ubpf_jit_support.c:172–204`
**API:** `ubpf_toggle_constant_blinding()` (`vm/inc/ubpf.h:150–170`)

Constant blinding is a JIT spray mitigation. When enabled (`vm->constant_blinding_enabled`),
all user-controlled immediate values are XORed with a random constant before emission.
At runtime, the original value is recovered by XORing again with the same random constant.

**Mechanism:**
```
compile-time: random = ubpf_generate_blinding_constant()
              blinded = imm ^ random
emit:         mov temp, blinded    ; Attacker-visible constant is meaningless
              xor temp, random     ; Recovers original value at runtime
              <operation> dst, temp
```

**Random generation** (`vm/ubpf_jit_support.c:172–204`): Platform-specific CSPRNG:
- Windows: `BCryptGenRandom()` with `BCRYPT_USE_SYSTEM_PREFERRED_RNG`
- Linux: `getrandom()` system call
- macOS: `arc4random_buf()`
- Fallback: `rand()` with thread-safe seeding

**Blinded instruction forms:** The following macros select blinded vs. normal emission:

| Macro | Defined at Line | Applies To |
|-------|----------------|------------|
| `EMIT_LOAD_IMM` | 698 | MOV64_IMM, LDDW, MUL/DIV/MOD immediate operand |
| `EMIT_ALU64_IMM32` | 1635 | ADD64_IMM, SUB64_IMM, OR64_IMM, AND64_IMM, XOR64_IMM |
| `EMIT_ALU32_IMM32` | 1644 | ADD_IMM, SUB_IMM, OR_IMM, AND_IMM, XOR_IMM, MOV_IMM |
| `EMIT_CMP_IMM32` | 1653 | All JMP_IMM comparisons (64-bit) |
| `EMIT_CMP32_IMM32` | 1662 | All JMP32_IMM comparisons (32-bit) |
| `EMIT_TEST_IMM32` | 1671 | JSET_IMM (64-bit) |
| `EMIT_TEST32_IMM32` | 1680 | JSET32_IMM (32-bit) |
| `EMIT_STORE_IMM32` | 1689 | STB, STH, STW, STDW (immediate stores) |

**Temporary register selection** (e.g., `vm/ubpf_jit_x86_64.c:508`):
- Default temp: R11
- If dst == R11: falls back to R10
- Both R10 and R11 are volatile (caller-saved) on all platforms

**ALU blinding detail** (`emit_alu64_imm32_blinded`, `vm/ubpf_jit_x86_64.c:499–539`):
```asm
; For: add dst, imm32
mov temp, (imm ^ random)     ; Load blinded constant
xor temp, random             ; Recover original in temp
add dst, temp                ; Apply operation using register form
```
The function converts the immediate-form opcode to a register-form opcode via a lookup table
(lines 520–529): `/0`→`01` (ADD), `/1`→`09` (OR), `/4`→`21` (AND), `/5`→`29` (SUB),
`/6`→`31` (XOR), `/7`→`39` (CMP).

### 5.2 Retpoline

**Source:** `vm/ubpf_jit_x86_64.c:1540–1578`
**Guard:** `#ifndef UBPF_DISABLE_RETPOLINES` (line 978)

The retpoline gadget mitigates Spectre v2 (Branch Target Injection) for indirect calls
to helper functions. It is emitted once at the end of the instruction stream and called
via `call retpoline_loc` instead of `call *rax`.

**Retpoline structure:**
```asm
retpoline_loc:
  call label1                  ; Push speculative return address
capture_ret_spec:
  pause                        ; F3 90 — spin hint for speculative execution
  jmp capture_ret_spec         ; Infinite loop (speculative path never escapes)
label1:
  mov [rsp], rax               ; Overwrite return address with actual target
  ret                          ; Return to actual target (rax)
```

**How it works:**
1. The `call label1` pushes the address of `capture_ret_spec` as the return address
2. At `label1`, the return address on the stack is overwritten with the actual target (RAX)
3. `ret` jumps to the actual target
4. The speculative execution engine, having predicted the `ret` would go to `capture_ret_spec`,
   enters the `pause; jmp` infinite loop and is thus contained

**When disabled** (line 984–996): The JIT emits a direct indirect call:
```asm
ff d0                          ; call *rax
```

### 5.3 W⊕X Memory Management

**Source:** `vm/ubpf_jit.c:104–167`

The JIT enforces Write XOR Execute (W⊕X) semantics to prevent the JIT buffer from being
simultaneously writable and executable:

1. **Translate phase:** Code is emitted into a `calloc`'d buffer (writable, not executable)
2. **Allocate executable memory:** `mmap(PROT_READ | PROT_WRITE, MAP_PRIVATE | MAP_ANONYMOUS)` (line 145)
3. **Copy:** `memcpy(jitted, buffer, jitted_size)` (line 151)
4. **Transition:** `mprotect(jitted, jitted_size, PROT_READ | PROT_EXEC)` (line 153)
5. **Free writable copy:** `free(buffer)` (line 162)

At no point is memory both writable and executable. The `calloc`'d buffer is freed after
copying, and the `mmap`'d region transitions from RW to RX atomically via `mprotect`.

## 6. Helper Function Dispatch

**Source:** `vm/ubpf_jit_x86_64.c:818–1008`

### 6.1 Static Table Dispatch

When no external dispatcher is registered (`dispatcher_loc` contains NULL), the JIT uses a
table of function pointers embedded at the end of the JIT'd code.

**Sequence** (lines 888–920):
```asm
; Load dispatcher address via RIP-relative load
mov rax, [rip + dispatcher_loc]      ; REX.W 8B 05 <rel32>

; Check if dispatcher is NULL
cmp rax, 0
jne external_dispatcher_path

; Default (static table) path:
mov eax, idx                         ; Helper function index
shl rax, 3                           ; Multiply by 8 (pointer size)
lea r10, [rip + helper_table_loc]    ; Load base of helper table
add rax, r10                         ; Calculate address in table
mov rax, [rax]                       ; Load function pointer

; Set 6th parameter (context/cookie):
; System V: mov r9, r11              ; Cookie goes in R9 (6th param)
; Windows:  mov [rsp], r11           ; Cookie spills to stack

; Call via retpoline (or direct):
call retpoline_loc                   ; Indirect call through RAX
```

**Helper table emission** (`emit_helper_table`, `vm/ubpf_jit_x86_64.c:1529–1537`):
Emits `MAX_EXT_FUNCS` (default 64) 8-byte function pointers as raw data.

### 6.2 Dynamic Dispatcher

When an external dispatcher is registered, the JIT routes through it instead:

**Sequence** (lines 925–961):
```asm
external_dispatcher_path:
; The dispatcher gets 7 arguments:
; Args 1-5: BPF R1-R5 (already in parameter registers)
; Arg 6: helper index
; Arg 7: context (cookie)

; System V:
mov r9, idx                          ; 6th param = index
; 7th param (cookie) is on stack from earlier push

; Windows:
; Spill cookie to [rsp+8]
; Spill index to [rsp+0]
; Push BPF R5 to stack (5th param on Windows)
; Allocate 32 bytes home register space

; Call via retpoline
call retpoline_loc
```

**Dispatcher function signature** (from `vm/inc/ubpf.h:223`):
```c
uint64_t (*external_function_dispatcher_t)(
    uint64_t p0, uint64_t p1, uint64_t p2, uint64_t p3, uint64_t p4,
    unsigned int index, void* cookie);
```

### 6.3 Parameter Marshaling

**BPF R1–R5 to x86-64 parameters:**

| BPF Reg | System V x86-64 | Windows x64 |
|---------|----------------|-------------|
| R1 (arg 1) | RDI (param 1) ✓ | R10 → needs `mov rcx, r10` |
| R2 (arg 2) | RSI (param 2) ✓ | RDX (param 2) ✓ |
| R3 (arg 3) | RDX (param 3) ✓ | R8 (param 3) ✓ |
| R4 (arg 4) | R10 → needs `mov rcx, r10` | R9 (param 4) ✓ |
| R5 (arg 5) | R8 (param 5) ✓ | R12 → spill to stack |

Before external calls, the JIT does:
```asm
mov rcx, r10                   ; Restore RCX from RCX_ALT (line 2268)
```
This is needed because RCX is reserved for shift operations during BPF execution but is
a parameter register in both ABIs.

**Context (cookie) parameter:**
The context pointer (first argument to the JIT'd function) is saved in `VOLATILE_CTXT` (R11)
at prologue time (line 1717). During helper dispatch:
- **System V:** 6th param goes in R9; 7th (cookie) goes on stack via the double-push of R11
  at lines 860–861
- **Windows:** Both 6th and 7th parameters spill to the stack above the home register space

**Stack frame around external calls** (lines 860–1007):
```asm
push r11                       ; Save context (also positions it for stack param)
push r11                       ; Maintain 16-byte alignment
; ... dispatch logic ...
; (Windows: sub rsp, 24 for extra stack params + alignment)
; (Windows: sub rsp, 32 for home register space before call)
; ... call ...
; (Windows: add rsp, 64 to clean up)
pop r11                        ; Restore context
pop r11                        ; Restore context
```

## 7. Local Function Calls

**Source:** `vm/ubpf_jit_x86_64.c:1473–1518`

Local function calls (BPF CALL with `src == 1`) implement callee-saved register preservation
and stack frame adjustment per RFC 9669.

**Sequence:**
```asm
; 1. Adjust R15 (BPF R10/frame pointer) down by current function's stack usage
sub r15, [rsp]                 ; 4C 2B 3C 24

; 2. Save callee-saved BPF registers (R6-R9)
push BPF_R6                   ; e.g., push rbx (System V)
push BPF_R7                   ; e.g., push r12
push BPF_R8                   ; e.g., push r13
push BPF_R9                   ; e.g., push r14

; 3. (Windows only) Allocate home register space
sub rsp, 32                   ; 4 × 8 bytes

; 4. Call target function
call target_pc                ; E8 <rel32> (patched later)

; 5. (Windows only) Deallocate home register space
add rsp, 32

; 6. Restore callee-saved BPF registers (reverse order)
pop BPF_R9
pop BPF_R8
pop BPF_R7
pop BPF_R6

; 7. Restore R15 by adding back current function's stack usage
add r15, [rsp]                ; 4C 03 3C 24
```

**Key invariant:** The top of the host stack (`[rsp]`) always holds the stack usage of the
currently-executing BPF function. The `sub r15, [rsp]` / `add r15, [rsp]` pair moves the
BPF frame pointer down/up by the caller's stack usage so the callee gets its own frame space.

**Stack frame structure per BPF function** (from `vm/ubpf_int.h:121–126`):
```c
struct ubpf_stack_frame {
    uint16_t stack_usage;
    uint16_t return_address;
    uint64_t saved_registers[4]; // R6-R9
};
```

**Fallthrough guard** (`vm/ubpf_jit_x86_64.c:1804–1813`):
If the instruction immediately before a local function entry can fall through to it,
the JIT emits a short jump (`EB xx`) to bypass the per-function prolog (stack usage push),
since the prolog should only execute when the function is entered via CALL.

## 8. Patchable Targets and Fixups

**Source:** `vm/ubpf_jit_support.h:44–117`, `vm/ubpf_jit_x86_64.c:2516–2625`

### 8.1 Patchable Target Types

During code emission, forward references cannot be resolved immediately. The JIT uses
`PatchableTarget` structures to record what needs to be patched:

**Special targets** (known locations, resolved to JIT state fields):

| Target | Resolved To | Usage |
|--------|-------------|-------|
| `Exit` | `state->exit_loc` | EXIT instruction, unwind-on-success |
| `Retpoline` | `state->retpoline_loc` | Indirect calls to helpers |
| `ExternalDispatcher` | `state->dispatcher_loc` | RIP-relative load of dispatcher ptr |
| `LoadHelperTable` | `state->helper_table_loc` | RIP-relative LEA of helper table |

**Regular targets** (resolved to BPF PC locations):

| Field | Meaning |
|-------|---------|
| `ebpf_target_pc` | BPF instruction index → looked up in `state->pc_locs[]` |
| `jit_target_pc` | Direct JIT offset (overrides PC lookup when non-zero) |
| `near` | If true, emit 8-bit relative offset; otherwise 32-bit |

### 8.2 Patchable Relative Tables

Four tables track unresolved references (`vm/ubpf_jit_support.h:149–155`):

| Table | Max Size | What It Tracks |
|-------|----------|---------------|
| `state->jumps` | `UBPF_MAX_INSTS` | All conditional/unconditional jumps and calls |
| `state->loads` | `UBPF_MAX_INSTS` | RIP-relative loads (dispatcher pointer) |
| `state->leas` | `UBPF_MAX_INSTS` | RIP-relative LEAs (helper table base) |
| `state->local_calls` | `UBPF_MAX_INSTS` | Local function call targets |

Each entry is a `struct patchable_relative` containing:
- `offset_loc`: Byte offset in the JIT buffer where the placeholder 4-byte offset was emitted
- `target`: A `PatchableTarget` describing where the reference should point

### 8.3 Resolution Process

**`resolve_patchable_relatives()`** (`vm/ubpf_jit_x86_64.c:2516–2625`):

**Jumps** (lines 2519–2569):
```
For each jump entry:
  target_loc = special ? {exit_loc | retpoline_loc}
             : jit_target_pc ? jit_target_pc
             : pc_locs[ebpf_target_pc]

  if near:
    rel8 = target_loc - (offset_loc + 1)    ; 1-byte offset
    assert(-128 <= rel8 < 128)
    buf[offset_loc] = (int8_t)rel8
  else:
    rel32 = target_loc - (offset_loc + 4)   ; 4-byte offset
    memcpy(&buf[offset_loc], &rel32, 4)
```

**Local calls** (lines 2571–2587):
```
rel32 = pc_locs[ebpf_target_pc] - (offset_loc + 4) - bpf_function_prolog_size
```
The `bpf_function_prolog_size` is subtracted because the `call` target needs to land
*before* the per-function prolog (which the called function executes as part of its entry).

**Loads** (lines 2589–2605):
```
target_loc = dispatcher_loc    ; Only ExternalDispatcher is valid
rel32 = target_loc - (offset_loc + 4)
```

**LEAs** (lines 2607–2623):
```
target_loc = helper_table_loc  ; Only LoadHelperTable is valid
rel32 = target_loc - (offset_loc + 4)
```

All relative offsets are computed as: `target - (source + sizeof(offset))`, which is the
standard x86 RIP-relative encoding where the offset is measured from the end of the
instruction.

## 9. Platform Differences (System V vs Win64)

| Aspect | System V ABI | Microsoft x64 ABI |
|--------|-------------|-------------------|
| **Parameter registers** | RDI, RSI, RDX, RCX, R8, R9 | RCX, RDX, R8, R9 |
| **Max register params** | 6 | 4 |
| **Non-volatile saved** | 6 (RBP, RBX, R12-R15) | 8 (RBP, RBX, RDI, RSI, R12-R15) |
| **Stack alignment push** | Extra 8B if even push count | Extra 8B if even push count |
| **BPF R1 mapped to** | RDI (= 1st param reg) | R10 (NOT a param reg) |
| **BPF R5 mapped to** | R8 (volatile) | R12 (non-volatile) |
| **Helper 6th param** | R9 (register) | Stack `[rsp+0]` |
| **Helper 7th param** | Stack (via pushed R11) | Stack `[rsp+8]` |
| **Home register space** | Not needed | 32 bytes allocated before every call |
| **5th param (helpers)** | R8 (register) | Stack (push before call) |
| **Local call overhead** | 4 pushes + call | 4 pushes + `sub rsp,32` + call + `add rsp,32` |
| **External call overhead** | 2 pushes + dispatch | 2 pushes + `sub rsp,24` + dispatch + `sub rsp,32` |

**Home register space:** Windows x64 ABI requires the caller to allocate 32 bytes (4 × 8)
of "shadow space" on the stack before every `call`, even if fewer than 4 parameters are passed.
This is emitted at:
- External helper calls: `vm/ubpf_jit_x86_64.c:975` and deallocated at line 1003
- Local function calls: `vm/ubpf_jit_x86_64.c:1496` and deallocated at line 1504
- Main prologue (before fake call): `vm/ubpf_jit_x86_64.c:1756`

**Practical impact:** JIT'd code for Windows is larger due to additional stack management
instructions. The BPF→x86 register mapping is also different, meaning the same BPF program
produces different machine code on each platform.
